# Writing Reference Catalogs

A reference doc is a **structured catalog of code-derived facts**. The reader treats it like an index — they Ctrl-F a name, land on a row, and see what they need.

## Properties of a good catalog

- **Tabular**: rows and columns, not prose paragraphs
- **Stable column set**: every entry has the same fields
- **Code-derived**: every fact in it can be regenerated by reading the code
- **Deduplicated**: each fact appears in exactly one place. If two catalogs need the same fact, one references the other.
- **Sortable** by an obvious key (alphabetical by name, by route, by file path)

## Common catalog types

### endpoints.md (HTTP API)

For each endpoint:

```markdown
### POST /api/bookings
Creates a new booking.
Auth required: Yes
Tenant header required: Yes

**Request**
\`\`\`json
{
  "staffId": "uuid",
  "serviceId": "uuid",
  "startsAt": "2026-04-15T10:00:00Z",
  "clientId": "uuid"
}
\`\`\`

**Response 201**
\`\`\`json
{
  "id": "uuid",
  "startsAt": "2026-04-15T10:00:00Z",
  "endsAt": "2026-04-15T10:30:00Z",
  "status": "Confirmed"
}
\`\`\`

**Errors**
| Code | Error | Cause |
|------|-------|-------|
| 409 | SLOT_TAKEN | Another booking exists for that slot |
| 422 | VALIDATION_FAILED | Required fields missing |
| 422 | NOT_WORKING_DAY | Staff not working that day |
```

Group by resource (Auth, Bookings, Staff, Clients, …). Top of file: base URL, common auth headers.

### entities.md (database)

For each entity:

```markdown
## Bookings
**Table**: `bookings`
**Inherits**: TenantEntity (adds `tenant_id`)

| Column | Type | Nullable | Constraints | Description |
|--------|------|----------|-------------|-------------|
| staff_id | uuid | No | FK Staff(id) | Assigned staff member |
| service_id | uuid | No | FK Services(id) | Booked service |
| client_id | uuid | No | FK Clients(id) | Booking client |
| starts_at | timestamptz | No | | Start time (UTC) |
| ends_at | timestamptz | No | | End time (UTC) |
| status | varchar(20) | No | DEFAULT 'Confirmed' | Confirmed, Cancelled, Completed, NoShow |

**Indexes**:
- `IX_Bookings_TenantId_StartsAt`
- `IX_Bookings_StaffId_StartsAt`

**Relationships**:
- BelongsTo Staff
- BelongsTo Service
- BelongsTo Client
```

Top of file: explain inheritance hierarchy (BaseEntity / TenantEntity / etc.) once, then catalog.

### components.md (UI components)

For each component:

```markdown
### `<Topbar>`

**File**: `src/components/layout/topbar.tsx`
**Used by**: `(authenticated)/layout.tsx`
**Directive**: `"use client"`

One-paragraph description of what it renders and when it's shown.

\`\`\`ts
type TopbarProps = {
  showTenantName?: boolean  // default true
  className?: string
}
\`\`\`

| Prop | Type | Default | Notes |
|---|---|---|---|
| showTenantName | boolean | true | Hide on tenant-less routes |
| className | string | — | Forwarded to root |

Behavioral notes: backdrop blur on scroll, sticky top-0, hides
on print media.
```

Group by folder (`brand/`, `layout/`, `ui/`).

### state.md (stores, query keys, contexts)

One section per store. Document:

- File path
- Storage backing (sessionStorage / localStorage / in-memory)
- State shape (typed)
- Actions
- Selector hooks (if any)
- When to use vs. not use
- SSR safety notes

Plus a query-keys section listing every key factory and what it caches.

### dependencies.md (cross-module calls)

For each module, what it depends on. Include a small ASCII graph at the top:

```text
DataContracts (no deps)
    ^
Utilities ──┐
    ^       │
Accessors ──┤
    ^       │
Engines ────┤
    ^       │
Managers ───┤
    ^       │
Api ────────┘
```

Then a per-module table:

```markdown
## BookingManager

| Depends on (Accessors) | Depends on (Engines) | Depends on (Managers) |
|---|---|---|
| Booking, Staff, Service, Client | Availability, Scheduling | — |
```

### tree.md (file tree)

Output of `tree -L N --gitignore src/` (or equivalent), pasted in a `text` code block. Regenerate every sync pass. Annotate top-level dirs:

```text
src/
├── app/                  Next.js App Router
│   ├── (auth)/          login, register, forgot, reset
│   ├── (authenticated)/ dashboard, calendar, bookings
│   └── onboarding/      tenant creation wizard
├── components/          reusable UI
├── lib/                 API client, query setup, i18n config
└── stores/              Zustand stores
```

## Catalog regeneration

Catalogs go stale fast. Two strategies:

1. **Manual regeneration during sync pass** — read code, regenerate catalog. Document the source of each table in a comment so the next person knows what to read.
2. **Generated** — script that emits the markdown from code (e.g. parsing OpenAPI for endpoints.md, EF migrations for entities.md). Better long-term but requires upfront work.

Either way, the file should start with a comment indicating:

```markdown
<!--
This file is regenerated during docs-sync passes.
Source: src/Rezervo.Api/Controllers/ + Scalar OpenAPI spec
Last regenerated: 2026-04-08
-->
```

## Common mistakes

- **Free-form prose where a table belongs.** "The bookings table has columns id, tenant_id, staff_id, service_id…" → table.
- **Scattered facts.** If endpoint X's request shape appears in both endpoints.md and a flow doc, one of them is wrong. Centralize in endpoints.md, link from the flow.
- **Stale examples.** When you change a request shape, update the catalog the same PR.
- **Missing the "why" — that goes in ADRs, not catalogs.** Catalogs are descriptive; ADRs are explanatory.
- **Forgetting indexes / constraints / relationships.** A schema catalog without indexes is half-useful.
